J'ai un pipeline ML trading EURUSD H1 dans /d:/Documents/learning-machine-learning. 
Triple barrier TP=30/SL=10 (ratio 3:1), commission=0.5p, slippage=1.0p 
→ friction totale 1.5p par trade. La cible actuelle marque "TP touché" 
comme win brut, ignorant la friction.

PROBLÈME : Le modèle apprend à viser des wins qui sont en réalité à peine 
breakeven après coûts. Cible mal alignée avec la profitabilité réelle.

TÂCHE : Créer une variante de cible "cost-aware" et comparer avec la cible 
classique.

CONCEPT :
- Cible classique : 1 si TP avant SL (LONG win), -1 si TP avant SL (SHORT 
  win), 0 sinon
- Cible cost-aware : 1 si (TP - friction > threshold_pips), idem pour 
  SHORT, sinon 0. Threshold suggéré : ≥ 2× friction = 3 pips de marge 
  réelle.

PLAN TECHNIQUE :
1. Modifier learning_machine_learning/features/triple_barrier.py :
   Ajouter `apply_triple_barrier_cost_aware(df, tp_pips, sl_pips, window, 
   pip_size, friction_pips, min_profit_pips=0.0) -> np.ndarray`
   - Si TP atteint mais (tp_pips - friction_pips) < min_profit_pips → label 0
   - Si SL atteint → label 0 ou direction perdante selon convention
   - Si timeout → calculer le PnL réel sur Close et labelliser selon 
     (pnl_réel - friction) > min_profit_pips

2. Ajouter un flag dans EurUsdConfig (instruments.py) :
   `cost_aware_labeling: bool = False` (off par défaut, opt-in)

3. Dans build_ml_ready (features/pipeline.py) : 
   - Si cfg.cost_aware_labeling → utiliser apply_triple_barrier_cost_aware
   - Sinon → comportement actuel
   - Logger la distribution des labels obtenue

4. Comparer dans ml_evolution.md :
   - Distribution des labels (classique vs cost-aware) — doit changer 
     significativement
   - Accuracy modèle (classique vs cost-aware)
   - WR backtest (devrait s'aligner sur le breakeven)
   - PnL net OOS

5. Tests unitaires :
   - test_cost_aware_zero_friction_equals_classic
   - test_cost_aware_high_friction_reduces_positive_labels
   - test_cost_aware_timeout_with_negative_pnl
   - test_cost_aware_threshold_filtering
